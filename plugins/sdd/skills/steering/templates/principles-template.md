# principles.md テンプレート

開発原則（TDD、SOLID、YAGNI等）を定義する。

```markdown
# 開発原則

## TDD（Test-Driven Development）

### 基本サイクル: Red-Green-Refactor

1. **Red（失敗するテストを書く）**
   - 実装前に期待する動作を定義
   - テストが失敗することを確認

2. **Green（テストを通す最小限の実装）**
   - テストを通すための最小限のコード
   - 完璧を求めず、まず動作を優先

3. **Refactor（コードを改善）**
   - テストが通る状態を維持しながら改善
   - 重複除去、命名改善、構造最適化

### TDDの実践ルール

- **型定義だけを先に作ることは禁止**
- **テストファースト原則**
- **小さなステップで進める**（各サイクル15分以内目安）

## SOLID原則

### Single Responsibility Principle（単一責任）

クラス・関数は単一の責任のみを持つ

### Open/Closed Principle（開放閉鎖）

拡張に対して開、修正に対して閉

### Liskov Substitution Principle（リスコフ置換）

派生クラスは基底クラスと置換可能

### Interface Segregation Principle（インターフェース分離）

使用しないメソッドへの依存を強制しない

### Dependency Inversion Principle（依存性逆転）

上位モジュールは下位モジュールに依存しない

### 落とし穴

- インターフェース作成が目的化してはいけない
- 実装が1つしかない場合は抽象化不要（YAGNI）

## YAGNI原則（You Aren't Gonna Need It）

- 実際に必要になるまで機能を追加しない
- 「将来使うかもしれない」で実装しない
- シンプルさを最優先

## [言語固有の原則]

### 型の扱い（TypeScriptの場合）

- `any`型の使用は禁止
- `interface`より`type`を優先
- 型推論を活用

### インポート/エクスポート

- Barrel import/exportは禁止
- 循環参照を避ける

## フェーズ管理原則（Phase-based Development）

### フェーズの独立性維持

**各フェーズで不要になったものはそのフェーズで消す**:

- フェーズ終了時に一時的なコード・コメントを削除
- 使用しなくなった関数・変数を即座に削除
- デバッグ用の出力を残さない

**事前のexport/importは禁止**:

- 後続のフェーズで使用するとしても、使用するフェーズで改めてexport/importする
- フェーズ間の不必要な依存を避ける
- 各フェーズは自己完結できるように設計する

### 理由

| 原則                 | 理由                                         |
| -------------------- | -------------------------------------------- |
| フェーズ終了時に削除 | コードベースをクリーンに保ち、可読性を維持   |
| 事前export禁止       | フェーズ間の結合度を下げ、変更コストを抑制   |
| 使用時点でexport     | 依存関係が明確になり、影響範囲が分かりやすい |

## コード品質基準

### 関数型プログラミング

- `map`、`filter`、`reduce`を優先
- 破壊的操作を避ける
- イミュータブルなデータ構造

### エラーハンドリング

- 例外は明示的にキャッチ
- 適切なエラーメッセージ

### セキュリティ

- SQLインジェクション対策
- XSS対策
- 認証・認可の適切な実装

## アンチパターン

| パターン         | 問題                         | 回避方法       |
| ---------------- | ---------------------------- | -------------- |
| 過度な抽象化     | 将来の要件を予測した過剰設計 | YAGNI原則      |
| 神オブジェクト   | 多くの責任を持つ巨大クラス   | 単一責任原則   |
| コピペ           | 同じコードの重複             | DRY原則        |
| マジックナンバー | ハードコードされた値         | 定数として定義 |
```

## 記載のポイント

- **プロジェクト固有にカスタマイズ**: 一般論ではなくプロジェクトでの適用方法
- **言語固有の原則**: 使用言語に応じたセクションを追加
- **アンチパターン**: よくある間違いを明示
